## P1 — Gates v2 features

### Caregiver UX exposure
**What:** Profile switcher in account screen, consent grant flow (mom approves daughter), notification routing per profile.
**Why:** Saudi caregiver-daughters (adult women managing aging parents' meds) are the highest-LTV cohort. v1 schema is ready (multi-profile + `consent_grants` collection). Just need UX.
**Pros:** Unlocks largest underserved cohort. Builds on v1 schema, no migration.
**Cons:** ~2 weeks of build. Requires PDPL legal review of consent flow copy.
**Context:** v1 ships single-user UX with multi-profile schema underneath. Schema locked in /plan-eng-review ARCH-1.2. PDPL compliance requires explicit consent records, already scaffolded.
**Depends on / blocked by:** v1 stable in production, PDPL legal sign-off on consent UX copy.

### Multi-store operational rollout
**What:** Inventory-per-branch UI, dispatch-per-branch routing, pharmacist routing UI, branch-aware admin dashboard.
**Why:** Estrogen brand expansion is the biggest leverage (CEO plan EXP2). Schema is multi-store-ready in v1; ops infrastructure isn't.
**Pros:** Each new store is a config change, not a re-platform. Scales the brand pan-KSA.
**Cons:** ~3 weeks of build, requires ops process change at the pharmacy.
**Context:** v1 Firestore schema has `branch_id` on every operational record (orders, products, inventory, pharmacists, drivers). Default value `main` in v1.
**Depends on / blocked by:** v1 stable, second store location decided.

## P2 — Adjacent features

### Insurance integration
**What:** Direct claims integration with Bupa Arabia, Tawuniya, MedGulf for prescription co-pay.
**Why:** No competitor has solved this well. Major differentiator. Reduces customer out-of-pocket.
**Pros:** Strong moat. Recurring B2B partnership revenue. Customer retention.
**Cons:** ~6 weeks of build per insurer. SAMA + insurer onboarding takes 2-4 months. Regulatory work.
**Context:** SAMA opened up health insurance digitization 2024-2025. Each insurer has its own claims API/format.
**Depends on / blocked by:** v1 stable, business development with at least one insurer.

### Wellness subscription boxes
**What:** Curated monthly boxes by life stage (postpartum kit, fertility kit, menopause kit, period-care subscription).
**Why:** Different product surface from chronic-refill subscriptions. Higher AOV. Higher engagement.
**Pros:** Net-new revenue stream. Bundles existing inventory creatively. Highly shareable on Instagram.
**Cons:** ~2 weeks of build once auto-refill subscription infra is shipped. Requires ops process for box assembly.
**Context:** Auto-refill subscription engine (v1.5) provides the recurring billing infrastructure. Wellness boxes are a different product type that reuses it.
**Depends on / blocked by:** v1.5 auto-refill subscription shipped, ops capacity for box assembly.

### Webhook-based catalog real-time sync
**What:** Existing online catalog pushes updates to a Cloud Function on every product change. Replace daily batch.
**Why:** Real-time stock-out prevention. Better customer experience for fast-moving inventory.
**Pros:** Eliminates "added to cart, then out-of-stock at fulfillment" frustration.
**Cons:** Requires existing catalog to support webhooks. ~1 week of build.
**Context:** v1 ships with daily batch sync via Cloud Scheduler. Upgrade only if stock-out becomes a customer complaint.
**Depends on / blocked by:** Existing catalog API supports webhooks (verify).

## P3 — Performance / cost / polish

### Typesense migration (Algolia replacement)
**What:** Replace Algolia with self-hosted Typesense on Cloud Run.
**Why:** Algolia bill grows with SKUs and users. Becomes meaningful at 50K+ SKUs or 100K+ users.
**Pros:** ~$30/mo Cloud Run vs Algolia's potentially-$2000+/mo at scale.
**Cons:** ~1-2 day migration, lose Algolia's polish. Self-hosted means self-monitored.
**Context:** Only triggers if Algolia bill becomes a real concern. Skip until then.
**Depends on / blocked by:** Algolia bill > $500/mo for 3 consecutive months.

### Native telemedicine video infrastructure
**What:** Replace FaceTime/Meet links in consult booking with in-app video (LiveKit, Daily, or Twilio Video).
**Why:** Better UX, retention inside the app, recording for compliance, audit trail.
**Pros:** Professional consult experience. Compliance-grade audit log.
**Cons:** ~2 weeks of build + MOH telemedicine licensing scope expansion. Per-minute video costs.
**Context:** v1.5 ships consult booking with FaceTime/Meet links. Native video is a v2.5+ upgrade.
**Depends on / blocked by:** Consult volume justifies (>100 consults/month), MOH telemedicine licensing complete.

### Driver dispatch route optimization
**What:** Route batching algorithm + dispatch UI for the 1-2 driver fleet.
**Why:** Driver currently routes manually. Optimization could 1.5-2x daily delivery capacity.
**Pros:** Defers hiring driver #3 by ~3 months. Reduces fuel costs.
**Cons:** ~1 week of build. May not matter until 100+ orders/day.
**Context:** v1 launches with 1 driver doing manual routing. Driver #2 and #3 hire when volume warrants.
**Depends on / blocked by:** Volume crosses 60+/day on multiple consecutive days.

### Image-optimization Cloud Function
**What:** On-the-fly resize and WebP conversion for product images via Cloud Function or imgix.
**Why:** Faster load times, less data for users on metered connections.
**Pros:** Better mobile UX, less Firebase Storage egress cost.
**Cons:** ~3 days of build, ongoing CPU cost.
**Context:** v1 uses original images from existing catalog. Only optimize if originals are large or load time becomes a complaint.
**Depends on / blocked by:** Image weight or load time becomes a real concern.

### Pharmacist-curated recommendations
**What:** "Pharmacist-recommended for you" cards in browse and product detail screens, populated by named pharmacists tagging products.
**Why:** Surfaces the moat (pharmacist expertise) inside the UX. Beats generic algorithmic recommendations.
**Pros:** Different from Nahdi's algorithmic recommendations. Builds on consult booking relationships.
**Cons:** ~1 week of build, requires pharmacist tooling for tagging.
**Context:** Customers said they don't trust call-center recommendations. Pharmacist-curated is the trusted alternative. Defer until v1.5 consult data accumulates.
**Depends on / blocked by:** v1.5 consult booking shipped + pharmacist tagging UI built.

### B2B / clinic white-label
**What:** Allow OB/GYN offices, fertility centers, dermatology clinics to white-label the app for their patients.
**Why:** Each clinic brings their patient base. Recurring B2B SaaS revenue.
**Pros:** Strong distribution leverage. Recurring revenue.
**Cons:** Major architectural work (multi-tenant config). 4-6 weeks at minimum.
**Context:** Premature for v1. Revisit at $5M+ ARR or 10K+ active users.
**Depends on / blocked by:** v1 + v1.5 stable, multi-store ops shipped, business model proven.
